# A2: Shell interpreter resolution and variable-as-command deny

**Avoid:** Don't add `bash`, `sh`, or `zsh` to safe-commands.txt. They are arbitrary execution vectors. The fix is to resolve through them to the script basename, not to whitelist the interpreter itself.

**Avoid:** Don't try to handle `bash -c 'command string'` specially. The `-c` flag is skipped as a flag, the next token (`'command string'`) becomes the "script" token, its basename won't match the safe list, so it denies by default. This is correct behavior.

**Avoid:** Don't try to resolve shell variables at hook time. The hook runs in an isolated process with no access to the agent's shell environment. Variable resolution would be fragile and incomplete.

**Avoid:** Don't put the variable detection inside `extract_command_words_from_segment`. The function's job is to extract command words; the deny-with-feedback logic belongs in the main verification loop where the JSON response is constructed.

## Steps

### Step 1: Add shell interpreter resolution in extract_command_words_from_segment

In `hooks/pretooluse-safe-commands.sh`, modify the `extract_command_words_from_segment` function. The change goes in the `else` branch (lines 163-166) that handles normal commands. After `outer_cmd` is set from `local word="${seg%% *}"` / `outer_cmd="${word##*/}"` (line 164-165) and before `[[ -n "$outer_cmd" ]] && cmds+=("$outer_cmd")` (line 168), add interpreter resolution.

When `outer_cmd` is a known shell interpreter (`bash`, `sh`, `zsh`), extract the first non-flag token from the remaining segment as the script path, then use its basename as `outer_cmd`:

```bash
    # Shell interpreter resolution: bash/sh/zsh script.sh -> script.sh
    if [[ "$outer_cmd" == "bash" || "$outer_cmd" == "sh" || "$outer_cmd" == "zsh" ]]; then
      local rest="${seg#"${seg%% *}" }"
      rest="${rest#"${rest%%[![:space:]]*}"}"
      local script_token=""
      while [[ -n "$rest" ]]; do
        local token="${rest%% *}"
        if [[ "$token" == "--" ]]; then
          rest="${rest#"$token"}"
          rest="${rest#"${rest%%[![:space:]]*}"}"
          if [[ -n "$rest" ]]; then
            script_token="${rest%% *}"
          fi
          break
        elif [[ "$token" == -* ]]; then
          rest="${rest#"$token"}"
          rest="${rest#"${rest%%[![:space:]]*}"}"
          continue
        else
          script_token="$token"
          break
        fi
      done

      if [[ -n "$script_token" ]]; then
        script_token="${script_token#\"}"
        script_token="${script_token%\"}"
        script_token="${script_token#\'}"
        script_token="${script_token%\'}"
        outer_cmd="${script_token##*/}"
      fi
      # If no script_token found (bare "bash"), outer_cmd stays "bash" -> fails safe check
    fi
```

This handles:
- `bash scripts/validate-plan` -> `validate-plan` (safe)
- `bash -e scripts/validate-plan` -> `validate-plan` (safe)
- `bash -euo pipefail scripts/validate-plan` -> `pipefail` (known limitation: `-o` takes an argument but flag-skipping doesn't know that; agents should use `set -euo pipefail` inside the script instead)
- `bash -- scripts/validate-plan` -> `validate-plan` (safe)
- `bash "$VAR"` -> `$VAR` (not safe -> deny path via variable detection below)
- `bash` (bare) -> `bash` (not safe -> fallthrough)
- `bash -c 'rm -rf /'` -> `-c` skipped, `'rm -rf /'` basename -> not safe -> fallthrough

### Step 2: Add variable-as-command detection in the main verification loop

Add a flag before the main verification loop (before `for seg in ...`, around line 207):

```bash
variable_as_command=0
```

Inside the inner loop, after `[[ -z "$word" ]] && continue` (line 212) and before the `is_safe` check (line 214), add variable detection:

```bash
    stripped="$word"
    stripped="${stripped#\"}"
    stripped="${stripped%\"}"
    stripped="${stripped#\'}"
    stripped="${stripped%\'}"
    if [[ "$stripped" == \$* ]]; then
      variable_as_command=1
      all_safe=0
      non_matching+=("$word")
      continue
    fi
```

### Step 3: Add deny-with-feedback response for variable-as-command

Replace the existing response block (lines 221-227):

```bash
if [[ $all_safe -eq 1 ]]; then
  printf '{"hookSpecificOutput":{"hookEventName":"PreToolUse","permissionDecision":"allow"}}\n'
else
  for nm in "${non_matching[@]}"; do
    printf '%s\n' "$nm" >> "$LOG_FILE"
  done
fi
```

With:

```bash
if [[ $all_safe -eq 1 ]]; then
  printf '{"hookSpecificOutput":{"hookEventName":"PreToolUse","permissionDecision":"allow"}}\n'
elif [[ $variable_as_command -eq 1 ]]; then
  printf '{"hookSpecificOutput":{"hookEventName":"PreToolUse","permissionDecision":"deny","permissionDecisionReason":"Command word is a shell variable — the safe commands hook cannot verify safety. Use the literal command/path instead of variable indirection."}}\n'
  for nm in "${non_matching[@]}"; do
    printf '%s\n' "$nm" >> "$LOG_FILE"
  done
else
  for nm in "${non_matching[@]}"; do
    printf '%s\n' "$nm" >> "$LOG_FILE"
  done
fi
```

The `permissionDecisionReason` is surfaced to the agent (not the user), enabling self-correction. The agent rewrites with the literal path, which the hook can then verify normally.
