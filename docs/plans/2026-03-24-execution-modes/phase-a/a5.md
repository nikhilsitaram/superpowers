# A5: Write dispatch-main.md

**Avoid:** Don't add parallelism — main context mode is intentionally sequential. The lead implements each task directly, one at a time, in dependency order. This is the simplest mode, designed for small plans (<=5 tasks) or debugging. Don't reference Agent tool, teammates, or background dispatch.

## Steps

### Step 1: Read dispatch-agent-teams.md and dispatch-subagents.md for structural reference

Read both files (created by A3 and A4) to understand the section structure. The main dispatch file follows the same lifecycle stages but everything happens in the lead's own context.

### Step 2: Create dispatch-main.md

Create `skills/orchestrate/dispatch-main.md`:

```markdown
# Dispatch: Main Context

No env vars required. No subagents or teammates. The lead implements each task directly in sequence.

## Execution Flow

Process tasks in dependency order (tasks with no unmet `depends_on` first). For each task:

### Implement

1. Read task metadata from plan.json and task prose from the task .md file
2. Mark task in-progress: `scripts/validate-plan --update-status plan.json --task {TASK_ID} --status in_progress`
3. Follow TDD for all implementation. **See:** `skills/orchestrate/tdd.md`
4. If the task consumes output from a prior task, write a boundary integration test
5. Implement exactly what the task specifies
6. Commit work

### Self-Review

After implementing, review your own work:
- Completeness: every requirement in the spec has corresponding code
- Quality: clear names, clean code, follows codebase conventions
- Testing: TDD followed, tests verify behavior not mocks
- Discipline: no overbuilding (YAGNI)

Fix any issues found.

### Completion

1. Write completion notes to `{PHASE_DIR}/{TASK_ID_LOWER}-completion.md`
2. Mark task complete: `scripts/validate-plan --update-status plan.json --task {TASK_ID} --status complete`

### Task Review

After each task completion, dispatch a reviewer (as a subagent, not a teammate):

```text
Agent(
  subagent_type: "general-purpose",
  model: "opus",
  prompt: "<substitute task-reviewer-prompt.md with task variables + diff range>"
)
```

Wait for the reviewer to complete (synchronous — no background dispatch in main mode). Extract `json review-summary`, triage issues, fix directly, repeat until review passes.

### Advance

After review passes:
1. Validate with `scripts/validate-plan --criteria plan.json --task {TASK_ID}`
2. Check if any blocked tasks are now unblocked via `scripts/validate-plan --check-deps`
3. Move to the next eligible task

## Handle Escalations

Since the lead is the implementer, Rule 4 violations are detected directly. Present to user immediately via AskUserQuestion.

## Review Feedback Loop

The lead fixes review findings directly — no agent dispatch needed for fixes. Read the reviewer's findings, apply fixes, re-run verification, re-dispatch reviewer until clean.
```

### Step 3: Verify simplicity

The main dispatch file should be the shortest of the three. It has no parallelism, no background dispatch, no mailbox, no worktree isolation. Verify it doesn't accidentally reference any of those concepts.

### Step 4: Commit

```bash
git add skills/orchestrate/dispatch-main.md
git commit -m "feat: add main context dispatch protocol for sequential execution"
```
