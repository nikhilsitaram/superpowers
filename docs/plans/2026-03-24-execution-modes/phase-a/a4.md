# A4: Write dispatch-subagents.md

**Avoid:** Don't emulate mailbox messaging — the Agent tool doesn't support it. Review fixes dispatch a NEW agent with original task context + reviewer findings. This costs more tokens (~2x for review-fix cycles) but avoids the complexity of capturing original agent state. This is a deliberate design decision documented in the design doc under Key Decision #6.

## Steps

### Step 1: Read dispatch-agent-teams.md for structural reference

Read `skills/orchestrate/dispatch-agent-teams.md` (created by A3) to understand the section structure. The subagents dispatch file follows the same outline but replaces teammate-specific mechanisms with Agent tool equivalents.

### Step 2: Create dispatch-subagents.md

Create `skills/orchestrate/dispatch-subagents.md`:

```markdown
# Dispatch: Subagents

No env vars required. Uses the Agent tool with `isolation: "worktree"` for parallel, worktree-isolated task execution.

## Prompt Templates

| Template | Purpose |
|----------|---------|
| `./implementer-prompt.md` | Task implementer (via Agent tool) |
| `./task-reviewer-prompt.md` | Per-task reviewer (via Agent tool) |

## Dispatch Implementer Agents

For each task with no unmet dependencies (verified via `scripts/validate-plan --check-deps`):

```text
Agent(
  subagent_type: "general-purpose",
  model: "opus",
  isolation: "worktree",
  run_in_background: true,
  prompt: "<substitute implementer-prompt.md template with task variables>"
)
```

Dispatch all eligible tasks simultaneously — `run_in_background` means the lead doesn't block on each agent.

## Process Completions (Push-Based)

When a background agent completes (push notification):

1. Read the agent's response for completion status
2. Read completion notes (`{PHASE_DIR}/{TASK_ID_LOWER}-completion.md`)
3. Dispatch a reviewer agent:
   ```text
   Agent(
     subagent_type: "general-purpose",
     model: "opus",
     isolation: "worktree",
     run_in_background: true,
     prompt: "<substitute task-reviewer-prompt.md with task variables + diff range>"
   )
   ```
4. When reviewer completes, extract the last `json review-summary` block
5. Triage issues: "fix" or "dismiss" (with reasoning)
6. If fixes needed: dispatch a NEW implementer agent with original task context + reviewer findings (the original agent is gone — no mailbox). The new agent reads the task .md, the completion notes, and the review findings, then applies fixes.
7. Repeat review cycle until review passes
8. Validate with `scripts/validate-plan --criteria plan.json --task {TASK_ID}`
9. **Incremental merge:** Merge this task's branch into the feature/integration branch
10. **Dependency gate:** Check blocked tasks. For each candidate, run `scripts/validate-plan --check-deps plan.json --task {TASK_ID}`. If all dependencies complete, dispatch a new implementer agent (worktree created from updated feature branch).

## Handle Escalations

Subagents cannot send mailbox messages. If a subagent encounters a Rule 4 violation (architectural change), it should document it in its completion notes and mark the task as needing escalation. The lead reads completion notes and presents escalations to the user.

## Review Feedback Loop

Unlike agent-teams, subagents are stateless — the original implementer agent is gone after completion. Review fixes require a NEW agent that receives:
- Original task metadata and prose
- Completion notes from the first implementation
- Reviewer findings with specific fix instructions
- The implementer's worktree (still exists on disk)

This fresh agent applies fixes in the existing worktree, then the review cycle repeats.
```

### Step 3: Verify structural alignment with dispatch-agent-teams.md

Both dispatch files should cover the same lifecycle stages:
- Dispatch mechanism
- Completion handling
- Review cycle
- Escalation handling
- Review feedback loop

The differences should be limited to:
- Agent tool vs teammate spawn
- run_in_background vs push-based idle notifications
- Fresh agent for fixes vs mailbox to original implementer
- Escalation via completion notes vs mailbox
