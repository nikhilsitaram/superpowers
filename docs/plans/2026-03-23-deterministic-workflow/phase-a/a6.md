# A6: Add review loop protocol to phase-dispatcher-prompt

**Avoid:** Don't add reviews.json writing to the phase dispatcher. The design doc states "Only the controlling context writes to reviews.json." The phase dispatcher parses the review-summary block and acts on it (fix/dismiss/re-review), but it reports findings back to orchestrate, which writes the record. Task-review records are for audit trail only.

**Avoid:** Don't change the dispatch model. Task-review stays in the phase dispatcher. The review loop protocol handles: parsing the JSON output, triaging issues, applying fixes, and triggering re-review when >5 actionable issues.

## Steps

### Step 1: Understand current task review flow

Read `skills/orchestrate/phase-dispatcher-prompt.md`. The current task review flow (steps 5-6 in "Your Process"):

```text
5. After implementer returns: dispatch task reviewer (./task-reviewer-prompt.md)
   - Issues found → dispatch new implementer to fix → re-review

6. Re-Review Gate: if reviewer found >5 issues, dispatch fresh reviewer after all fixes
```

This needs to become: parse review-summary → triage → fix → conditional re-review with iteration cap.

### Step 2: Replace steps 5-6 with review loop protocol

Replace the current steps 5 (after implementer) and 6 (re-review gate) with:

```text
5. **After implementer returns: run task review loop**
   a. Dispatch task reviewer (`./task-reviewer-prompt.md`) with TASK_BASE_SHA..HEAD
   b. Extract the last `json review-summary` fenced block from the reviewer response
      - If the block is missing or malformed JSON → treat as verdict:fail, dispatch a fresh reviewer (this prevents silent skipping — a missing summary means re-review, not pass)
   c. Triage each issue in the `issues` array:
      - "fix" → dispatch implementer to fix
      - "dismiss" → document reasoning (will be included in report to orchestrate)
   d. Count actionable (non-dismissed) issues:
      - 0 actionable → proceed to step 6
      - 1-5 actionable → fix all, verify fixes, proceed to step 6
      - >5 actionable → fix all, dispatch fresh reviewer (back to 5a)
      - Max 3 iterations. After 3rd iteration with >5 issues → stop and report to orchestrate context for user escalation
   e. Report review results to orchestrate: issues_found, severity counts, dismissed (with reasons), fixed count, verdict
```

### Step 3: Update the "Report Back" section

Update the Report Back section to include review data that orchestrate needs for reviews.json:

```text
## Report Back

Return to orchestrate context:
- Tasks completed: [list with task IDs]
- HEAD SHA: `git rev-parse HEAD`
- Integration test status
- Deviations (if any)
- Per-task review summaries: for each task, include the final review-summary JSON and dismissal reasoning
- Any concerns for the orchestrate context
```

Read through the full updated `phase-dispatcher-prompt.md` to verify:
- Step numbering is consistent (steps should be numbered 1-11 sequentially)
- No contradictions between the new review protocol and existing deviation rules
- The "dispatch fresh reviewer" instruction is clear about using a new subagent (not the same one)
