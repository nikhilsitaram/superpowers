# A2: Add conditional merge strategy to merge-pr SKILL.md

**Avoid:** Don't make the conditional logic depend on reading plan.json at merge time inside merge-pr SKILL.md. The merge-pr skill is a standalone skill that doesn't know about plan.json. Instead, the orchestrator passes the merge strategy as context — merge-pr simply uses `--rebase` or `--squash` based on what the caller specifies. The merge-pr SKILL.md documents both strategies and when each applies.

**Avoid:** Don't change phase PR merge behavior. Phase PRs (base is `integrate/*`) always use `--squash`. Only the final PR (`integrate/*` to main) uses the conditional strategy.

## Steps

Read `skills/merge-pr/SKILL.md`. The current Step 3 (Merge) on lines 46-58 has:

```text

```

The H1 description on line 8 says "Confirm, squash merge, and clean up branches and worktrees."

### Step 2: Add merge strategy parameter

Update Step 3 (Merge) to support a merge strategy flag. Replace the hardcoded `--squash` with a conditional:

```text
### Step 3: Merge

**Pre-merge rebase check:** (unchanged)

**Merge strategy:**
- Default: `gh pr merge $PR_NUMBER --squash`
- When caller passes `--rebase` (orchestrate does this for multi-phase final PRs): `gh pr merge $PR_NUMBER --rebase`

Multi-phase plans produce one commit per phase on the integration branch (from squash-merging each phase PR). Rebase preserves this per-phase history on main. Single-phase plans use squash (one phase = one commit).
```
