# A5: Run skill-eval for orchestrate and merge-pr

**Avoid:** Don't skip evals because "the changes are just markdown." Skill files are the codebase — untested skill edits silently teach wrong behavior. The polling gate and merge strategy are behavioral changes that need eval coverage.

**Avoid:** Don't create evals that test surface-level keyword presence (e.g., "output mentions polling"). Write evals that test behavioral outcomes: does the agent correctly describe the poll-review-merge sequence when asked to orchestrate a multi-phase plan? Does it use --rebase for multi-phase final PRs?

## Steps

### Step 1: Resolve eval paths

```bash
PLUGIN_ROOT=$(python3 -c "
import json, os
p = json.load(open(os.path.expanduser('~/.claude/plugins/installed_plugins.json')))
print(p['plugins']['claude-caliper@claude-caliper'][0]['installPath'])
")
EVAL_ROOT=~/.claude/skill-evals
SKILL_EVAL_SCRIPT=$PLUGIN_ROOT/skills/skill-eval/scripts/run_eval.py
AGGREGATE_SCRIPT=$PLUGIN_ROOT/skills/skill-eval/scripts/aggregate_benchmark.py
```

### Step 2: Snapshot before versions

```bash
mkdir -p $EVAL_ROOT/orchestrate/snapshot-before
cp skills/orchestrate/SKILL.md $EVAL_ROOT/orchestrate/snapshot-before/SKILL.md
```

Wait — the implementer has already modified the SKILL.md files in tasks A1-A3. The "before" snapshot needs to be taken from git:

```bash
mkdir -p $EVAL_ROOT/orchestrate/snapshot-before $EVAL_ROOT/merge-pr/snapshot-before
git show HEAD:skills/orchestrate/SKILL.md > $EVAL_ROOT/orchestrate/snapshot-before/SKILL.md
git show HEAD:skills/merge-pr/SKILL.md > $EVAL_ROOT/merge-pr/snapshot-before/SKILL.md
```

Use `HEAD` because the current commit is the pre-edit state (A1-A4 changes are uncommitted or on the working branch — the snapshot should be from before those commits).

### Step 3: Create orchestrate evals.json

Create `$EVAL_ROOT/orchestrate/evals.json` with 2-3 scenarios:

```json
{
  "skill_name": "orchestrate",
  "evals": [
    {
      "id": 1,
      "name": "multi-phase-poll-review-gate",
      "prompt": "I have a 3-phase plan (phases A, B, C where B and C depend on A). Walk me through exactly what happens after phase A's create-pr step completes. Include the check polling behavior, review-pr invocation, and what happens on timeout.",
      "expectations": [
        "Describes polling gh pr checks every 60 seconds",
        "Mentions review_wait_minutes from plan.json with default of 10",
        "Invokes review-pr to address external reviewer comments",
        "On timeout proceeds with warning rather than blocking",
        "Does not re-poll after review-pr pushes fixes",
        "Merges with gh pr merge --squash for the phase PR",
        "Updates integration worktree with git pull after merge"
      ]
    },
    {
      "id": 2,
      "name": "final-pr-merge-strategy",
      "prompt": "I have a completed 2-phase plan using the merge-pr workflow. All phases are done. Describe the final PR flow from integrate/my-feature to main, including the merge strategy and why.",
      "expectations": [
        "Creates final PR from integrate/my-feature to main",
        "Polls checks on the final PR same as phase PRs",
        "Invokes review-pr on the final PR",
        "Uses --rebase merge for the multi-phase final PR",
        "Explains rebase preserves per-phase commit history",
        "Invokes merge-pr with --rebase flag"
      ]
    },
    {
      "id": 3,
      "name": "single-phase-final-pr",
      "prompt": "I have a single-phase plan with the merge-pr workflow. Phase A is complete. What merge strategy is used for the final PR to main and why?",
      "expectations": [
        "Uses --squash merge for single-phase final PR",
        "Explains single phase has no phase history worth preserving",
        "Still polls checks and invokes review-pr before merging"
      ]
    }
  ]
}
```

### Step 4: Create merge-pr evals.json (or update existing)

Check `$EVAL_ROOT/merge-pr/evals.json`. If it exists, add a scenario for the `--rebase` flag. If not, create it:

```json
{
  "skill_name": "merge-pr",
  "evals": [
    {
      "id": 1,
      "name": "rebase-merge-strategy",
      "prompt": "I'm merging a PR from integrate/auth-feature to main. The plan had 3 phases. Should I use --squash or --rebase and why?",
      "expectations": [
        "Recommends --rebase for multi-phase integration branch PR",
        "Explains rebase preserves per-phase commit history on main",
        "Notes that each phase was already squash-merged into the integration branch"
      ]
    },
    {
      "id": 2,
      "name": "standard-squash-merge",
      "prompt": "I have a PR from my feature branch to main. It's been reviewed. Merge it.",
      "expectations": [
        "Uses --squash as the default merge strategy",
        "Runs pre-merge rebase check",
        "Cleans up branch and worktree after merge"
      ]
    }
  ]
}
```

### Step 5: Write config.json and run evals

Create `$EVAL_ROOT/orchestrate/iteration-1/config.json` and `$EVAL_ROOT/merge-pr/iteration-1/config.json` per the skill-eval schema.

Run evals for both skills (both before and after variants, 3 runs each). Spawn all four eval runs simultaneously for efficiency.

### Step 6: Grade and aggregate

Dispatch grader subagents for each eval. Then run the aggregate script to produce benchmark.md for each skill.

### Step 7: Analyze results

Review benchmark.md for each skill. After variant pass rate should be >= before variant. If regressions found, investigate the specific failing assertions and fix the SKILL.md edits from A1/A2 before proceeding.

### Step 8: Commit eval artifacts

```text
test: add skill-evals for orchestrate poll-review gate and merge-pr rebase strategy
```
