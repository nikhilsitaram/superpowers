## Context

Static `depends_on` declarations are a planning-time guess. Implementation often creates connections the planner didn't anticipate. The orchestrator needs a reconciliation step between phases to detect and bridge these gaps.

## What to Do

Add a Dependency Reconciliation sub-step to the per-phase execution flow in `skills/orchestrate/SKILL.md`. Insert it between phase completion processing (rebase, ship, merge) and dispatching newly-ready phases — after extracting `PHASE_TASKS_JSON` / `PRIOR_COMPLETIONS` / `CROSS_PHASE_HANDOFF_TARGETS` and before dispatching the phase dispatcher.

### Reconciliation behavior

For each phase about to be dispatched (skip phases with empty `depends_on`):

1. For each completed dependency phase, run `git diff --name-only PHASE_BASE_SHA..phase-{letter}` to get files actually touched
2. Read completion notes (already in `PRIOR_COMPLETIONS`)
3. Detect impacts against current phase's tasks:
   - **File overlap:** Files in the dependency diff that also appear in a current-phase task's `files.modify` or `files.create` in plan.json
   - **Semantic impact:** Completion notes mention new exports, APIs, config keys, or deviations — and a current-phase task's `done_when` suggests it consumes or interacts with these
4. Filter out declared dependencies — skip impacts where the affected task already has the source task in its `depends_on`
5. For each undeclared impact, insert into the affected task .md file after the H1 header and any existing `## Handoff from` sections:
   ```markdown
   ## Reconciliation: Impact from Phase {LETTER}

   **Source:** {task ID} — {what changed}
   **Files:** {overlapping file paths}
   **Context:** {why this matters for this task}
   ```
6. Log: `"Reconciliation: injected ad-hoc handoff notes for {task IDs} based on Phase {LETTER} changes"`
