**Avoid:** Don't attempt automatic conflict resolution beyond what `git rebase` handles cleanly. If conflict markers appear, pause for the user — the agent guessing at conflict resolution in skill-level code will produce subtle bugs.

## Steps

### Step 1: Add rebase step to the post-completion flow

In the serialized completion processing (step 4c/d from orchestrate), add a rebase step after implementation review and before shipping the phase PR:

```text
- Rebase phase on latest integration:
  cd .claude/worktrees/<feature>-phase-{letter}
  git fetch origin integrate/<feature>
  git rebase origin/integrate/<feature>
```

### Step 2: Add conflict resolution heuristic

After the rebase step:

```text
- If git rebase succeeds (no conflict markers): run tests to verify, continue to ship
- If git rebase fails (conflict markers present): abort rebase, pause, present both
  versions to the user as a Rule 4-style escalation
```

Note: for the first phase to merge in a wave, rebase is a no-op (integration hasn't changed). The rebase only matters for the second+ parallel phase to complete.

### Step 3: Document when rebase is relevant

Add a brief note explaining: "Rebase is needed when parallel phases both modify files on the integration branch. The first phase to merge updates integration; the second phase must rebase to incorporate those changes. If phases are truly independent (no file overlap), rebase succeeds cleanly."
