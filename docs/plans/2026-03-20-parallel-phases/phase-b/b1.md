# B1: Add DAG wave logic for parallel phase dispatch to orchestrate

**Avoid:** Don't implement the rebase-before-merge logic here — that's B2. This task adds the DAG construction and wave-based dispatch loop. Keep sequential fallback clean: a fully linear plan (A→B→C) should produce the same behavior as Phase A's sequential model.

## Steps

### Step 1: Add DAG construction section

Add to orchestrate SKILL.md a section describing how to build the phase dependency graph from plan.json:

```bash
# Extract phase letters and their depends_on
jq -r '.phases[] | "\(.letter):\(.depends_on | join(","))"' plan.json
```

Build an adjacency list. Identify the initial wave: phases whose `depends_on` is empty or contains only completed phases.

### Step 2: Replace sequential loop with wave-based dispatch

Replace the "For each phase" sequential loop with:

```text
LOOP until all phases complete:
  a. Identify ready phases: depends_on all satisfied (completed)
  b. For each ready phase IN PARALLEL:
     - Create worktree
     - Dispatch phase dispatcher as Agent (one per ready phase)
  c. As each phase returns (serialized processing):
     - Implementation review
     - Rebase + merge (see B2)
     - Ship phase PR
     - Mark phase complete
  d. Repeat: check for newly ready phases
```

Emphasize that step (c) is serialized — one phase processed at a time even when multiple finish concurrently — to prevent race conditions on the integration branch.
