# A2: Update test fixture for new schema fields

> **Handoff from A1:** The `do_schema()` function in `scripts/validate-plan` now requires two new fields: (1) top-level `"workflow"` field -- must be one of `"ship"`, `"review-only"`, `"plan-only"` (emits `missing_field: workflow` or `invalid_workflow`); (2) `"depends_on"` array on each phase object -- must contain only letters of earlier phases (emits `missing_field: phases[N].depends_on` or `invalid_depends_on`). The test_schema.sh Test 1 already patches the fixture with these fields via jq in a temp copy. The fixture file itself still lacks them -- that's what this task adds. After adding, the raw fixture path can pass `--schema` directly.

**Avoid:** Don't change the fixture structure beyond adding the new fields — the render tests compare against `plan.md` which is generated from the fixture. If the fixture changes structurally, `test_render.sh` will break. Only add `workflow` and `depends_on` fields.

## Steps

### Step 1: Add workflow and depends_on to the valid fixture

Add `"workflow": "ship"` to `tests/validate-plan/fixtures/valid-plan/plan.json` at the top level (after `"status"`). Add `"depends_on": []` to Phase A and `"depends_on": ["A"]` to Phase B. This ensures the fixture passes the stricter schema validation added in A1.

### Step 2: Verify schema passes

```bash
bash scripts/validate-plan --schema tests/validate-plan/fixtures/valid-plan/plan.json
```

Must exit 0.

### Step 3: Update expected plan.md if render output changed

Run render and compare:

```bash
bash scripts/validate-plan --render tests/validate-plan/fixtures/valid-plan/plan.json
```

If `--render` now includes `workflow` in its output, update the expected `plan.md` fixture to match. If render doesn't output workflow (it's metadata, not prose), no change needed.

### Step 4: Run full test suite

```bash
bash tests/validate-plan/test_schema.sh
bash tests/validate-plan/test_render.sh
bash tests/validate-plan/test_e2e.sh
```

All must pass. Commit.
