# A1: Add workflow and phase depends_on validation to validate-plan

**Avoid:** Don't add workflow/depends_on as optional fields that silently pass when missing — they should be required. The whole point is to make the signal explicit and validated. However, keep backward compatibility for plans without `workflow` by defaulting to `"ship"` only during `--update-status` and `--criteria` modes, not during `--schema` (schema must be strict).

## Steps

### Step 1: Write failing tests for workflow validation

Add tests to `tests/validate-plan/test_schema.sh` following the existing pattern (setup temp dir, write plan.json, run validate-plan --schema, assert exit code and stderr content):

1. **Test: valid workflow values pass** — plan.json with `"workflow": "ship"` passes schema
2. **Test: invalid workflow value fails** — `"workflow": "auto"` produces error `invalid_workflow`
3. **Test: missing workflow field fails** — no workflow key produces error `missing_field: workflow`

Run tests, verify they all FAIL (script doesn't check workflow yet).

### Step 2: Implement workflow validation in validate-plan

In `scripts/validate-plan`, inside the `do_schema()` function, after the existing top-level field checks (around the `schema`, `status`, `goal`, `architecture`, `tech_stack` block):

1. Add a check that `workflow` field exists: `jq -e '.workflow' "$PLAN" > /dev/null 2>&1` — if missing, emit `missing_field: workflow`
2. Add an enum check: workflow must be one of `ship`, `review-only`, `plan-only` — if invalid, emit `invalid_workflow: <value>`

Run tests, verify the 3 new workflow tests PASS and all existing tests still pass.

### Step 3: Write failing tests for phase depends_on validation

Add tests to `tests/validate-plan/test_schema.sh`:

1. **Test: valid depends_on passes** — phase B with `"depends_on": ["A"]` passes
2. **Test: depends_on references non-existent phase fails** — `"depends_on": ["Z"]` produces error containing `invalid_depends_on`
3. **Test: circular dependency detected** — Phase A depends on B, Phase B depends on A produces error `circular_dependency`

Run tests, verify they FAIL.

### Step 4: Implement phase depends_on validation

In `do_schema()`, inside the phase validation loop:

1. Check that each phase object has a `depends_on` array: `jq -e ".phases[$i].depends_on" "$PLAN"` — if missing, emit `missing_field: depends_on` for that phase
2. For each entry in depends_on, verify it references a phase letter that appears earlier in the phases array (same forward-reference check as task depends_on but at phase level) — if not, emit `invalid_depends_on: Phase {letter} references {ref}`
3. After all phases are validated, run a cycle detection pass: build adjacency list from phase depends_on, do a topological sort (DFS with visited/in-stack tracking), emit `circular_dependency` if a back-edge is found

Run tests, verify all new + existing tests pass.

### Step 5: Update the valid fixture to include new fields

Add `"workflow": "ship"` to `tests/validate-plan/fixtures/valid-plan/plan.json` at the top level. Add `"depends_on": []` to Phase A and `"depends_on": ["A"]` to Phase B. This ensures the fixture remains valid under the stricter schema.

Run the full test suite: `bash tests/validate-plan/test_schema.sh && bash tests/validate-plan/test_render.sh && bash tests/validate-plan/test_e2e.sh`

All must pass. Commit.
