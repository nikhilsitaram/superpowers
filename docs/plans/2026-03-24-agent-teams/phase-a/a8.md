# A8: Update task-reviewer-prompt.md for teammate model

**Avoid:** Don't change the 6-point checklist or the output format — those are working well. The changes are structural: how the reviewer is dispatched (teammate, not subagent of phase dispatcher) and what context it receives.

## Steps

### Step 1: Read current task-reviewer-prompt.md

Read `skills/orchestrate/task-reviewer-prompt.md` end-to-end. Note:
- Currently dispatched as `Agent tool` by phase dispatcher
- Receives `{IMPLEMENTER_REPORT}` from the dispatcher (who read the implementer's output)
- Git range is within a shared worktree (`{BASE_SHA}..{HEAD_SHA}`)
- No reference to per-task completion files

### Step 2: Update dispatch model

Change the dispatch model from `Agent tool` to teammate spawn. The reviewer teammate is spawned by the lead (orchestrator), not by a phase dispatcher.

Update the header comment:
```markdown
# Task Reviewer Prompt Template

Dispatch a reviewer teammate to evaluate a single task's implementation. Spawned by the lead when an implementer teammate finishes.
```

### Step 3: Update context sources

The reviewer no longer receives `{IMPLEMENTER_REPORT}` from a dispatcher intermediary. Instead:

1. Add `{TASK_COMPLETION_FILE}` variable — path to `{PHASE_DIR}/{task_id_lower}-completion.md`, which the implementer teammate wrote.

2. Replace the "Implementer Report" section:

```text
  ## Implementer Completion Notes

  Read {TASK_COMPLETION_FILE} for the implementer's self-reported summary,
  deviations, files changed, and test results.

  Do not trust the notes at face value. Verify every claim by reading
  actual code. The notes are a starting point, not evidence.
```

### Step 4: Update git range context

The reviewer now gets the task's branch diff rather than a range within a shared worktree. Update:

```text
  ## Git Range

  The task was implemented in {REPO_PATH}

  ```bash
  git diff --stat {BASE_SHA}..{HEAD_SHA}
  git diff {BASE_SHA}..{HEAD_SHA}
  ```

  Read every file in the diff.
```

Variables:
- `{REPO_PATH}` — the implementer teammate's worktree path
- `{BASE_SHA}` — the SHA before the task started (recorded by lead when teammate was spawned)
- `{HEAD_SHA}` — current HEAD of the teammate's worktree (read by lead from teammate output)

### Step 5: Update variables list

Add a variables block at the top (matching the implementer prompt pattern):

```markdown
**Variables:**
- `{TASK_ID}` — the task ID (e.g., A1)
- `{TASK_SPEC}` — task metadata + prose combined
- `{TASK_COMPLETION_FILE}` — path to implementer's completion notes
- `{REPO_PATH}` — implementer teammate's worktree path
- `{BASE_SHA}` — SHA before task started
- `{HEAD_SHA}` — SHA after task completed
```

### Step 6: Verify the 6-point checklist is unchanged

Confirm sections 1-6 (Spec Fidelity, TDD Discipline, Test Quality, Code Correctness, Security, Simplicity) and the output format (Issues Found, Assessment, Review Summary) are unchanged.
