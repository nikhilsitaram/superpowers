### Step 2: Write the new SKILL.md

The new structure:

```markdown
---
name: orchestrate
description: Use when executing implementation plans with independent tasks in the current session

# Orchestrate

Execute plans via agent teams. Phases run sequentially; tasks within each phase run in parallel as teammates. Push-based idle notifications replace polling.

**Requires:** `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` environment variable. Check at start — abort with clear message if not set.

## Prompt Templates

| Template | Purpose |
|----------|---------|
| `./implementer-prompt.md` | Task implementer teammate |
| `./task-reviewer-prompt.md` | Per-task reviewer teammate |
| `skills/implementation-review/reviewer-prompt.md` | Cross-task reviewer (orchestrate context) |

## Progress Tracking

TaskCreate per phase: "Execute phase {LETTER} tasks ({N})", "Implementation review phase {LETTER}", "Create PR". Final: "Mark plan complete". Set `addBlockedBy`. Mark `in_progress` / `completed` as you go.

## Setup

Before first phase:
- Verify environment: `[ "$CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS" = "1" ] || { echo "Agent teams required"; exit 1; }`
- Read workflow: `WORKFLOW=$(jq -r '.workflow' plan.json)`
- Count phases: `PHASE_COUNT=$(jq '.phases | length' plan.json)`
- `scripts/validate-plan --schema plan.json` — abort if structural errors (catches file-set overlap, ID mismatches)
- `scripts/validate-plan --update-status plan.json --plan --status "In Development"`
- `PLAN_BASE_SHA=$(git rev-parse HEAD)`
- `PLAN_DIR=$(dirname "$(realpath plan.json)")` and `[ -f "$PLAN_DIR/reviews.json" ] || echo '[]' > "$PLAN_DIR/reviews.json"`
- Push branch: `git push -u origin HEAD`

## Per-Phase Execution (Sequential)

For each phase in order (A, B, C...):

### 1. Prepare phase

- `PHASE_DIR=${PLAN_DIR}/phase-{letter_lower}`
- `PHASE_BASE_SHA=$(git rev-parse HEAD)`
- For multi-phase: create integration branch `integrate/<feature>` if not exists; create phase worktree from it
- For single-phase: work directly in feature worktree
- **Bootstrap dependencies** in the worktree. **See:** skills/design/dependency-bootstrap.md
- For non-first phases: write cross-phase handoff notes into task .md files (read prior phase completion files, append handoff section with function signatures, file paths, config keys, APIs created)

### 2. Spawn implementer teammates (parallel)

For each task in the phase:
- Read task metadata from plan.json and task prose from `{PHASE_DIR}/{task_id_lower}.md`
- Spawn one implementer teammate per task using `./implementer-prompt.md` template
- Each teammate gets an auto-provisioned worktree and works independently
- All teammates execute concurrently (file-set isolation guarantees no conflicts)

### 3. Process completions (push-based)

As each implementer teammate finishes (idle notification — no polling):
- Read the teammate's task completion notes from `{PHASE_DIR}/{task_id_lower}-completion.md`
- Spawn a reviewer teammate for the completed task using `./task-reviewer-prompt.md`
- Reviewers run concurrently with still-active implementers

### 4. Handle escalations

Teammates that hit unresolvable issues send a structured message to the lead via mailbox. On receipt:
- Respond with guidance if the issue is solvable
- Stop the teammate and reassign if needed
- Rule 4 (architectural change): escalate to user via AskUserQuestion

### 5. Phase wrap-up

When all implementers + reviewers for the phase are done:
- Merge all task branches into the feature/integration branch (no conflicts due to file-set isolation)
- Dispatch implementation-review with `PHASE_BASE_SHA..HEAD`, run Review Loop Protocol (scope: `phase-{letter_lower}`)
- `scripts/validate-plan --check-review plan.json --type impl-review --scope phase-{letter_lower}`
- Run phase criteria: `scripts/validate-plan --criteria plan.json --phase {LETTER}`
- Update status: `scripts/validate-plan --update-status plan.json --phase {LETTER} --status "Complete (YYYY-MM-DD)"`

For multi-phase:
- Create phase PR: invoke create-pr with `--base integrate/<feature>`
- External review gate (skip if `review_wait_minutes` is 0): poll `gh pr checks` every 60s, max wait from plan.json
- Review feedback: invoke review-pr --automated
- Merge phase PR: `gh pr merge --squash`, pull into integration worktree
- Clean up phase worktree

## Review Loop Protocol

(Keep existing — unchanged from current SKILL.md)

After each impl-review dispatch:
1. Extract last `json review-summary` fenced block. Missing/malformed → verdict:fail, re-dispatch.
2. Triage issues: "fix" (dispatch implementer) or "dismiss" (with reasoning)
3. actionable == 0 → write reviews.json record verdict:pass, advance
4. actionable 1-5 → fix all, verify, write record verdict:pass, advance
5. actionable > 5 → fix all, write record verdict:fail, re-dispatch (max 3, then escalate)

## Single-Phase Plans

Skip integration branch. Work directly in feature worktree. Execute phase A per steps above (without phase PR/merge). Then:
1. Run plan criteria: `scripts/validate-plan --criteria plan.json --plan`
2. `scripts/validate-plan --update-status plan.json --plan --status Complete`
3. Route on workflow (create-pr or merge-pr)

## After All Phases (Multi-Phase Only)

1. Run plan criteria: `scripts/validate-plan --criteria plan.json --plan`
2. Final review: implementation-review with `PLAN_BASE_SHA..HEAD`, Review Loop Protocol (scope: `final`)
3. `scripts/validate-plan --check-review plan.json --type impl-review --scope final`
4. `scripts/validate-plan --update-status plan.json --plan --status Complete`
5. Route on workflow

## Key Constraints

| Constraint | Why |
|------------|-----|
| Record PLAN_BASE_SHA before first phase | Final cross-phase review needs total diff |
| Record PHASE_BASE_SHA per phase | Per-phase review needs exact phase start |
| Use validate-plan for all status updates | Keeps plan.json and plan.md in sync |
| Check CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS | No fallback — agent teams is the only execution model |
| Run --schema before execution | Catches file-set overlap and structural errors early |

## Integration

**Workflow:** design → draft-plan → **this skill** → create-pr → review-pr → merge-pr
**See:** `tdd.md` — TDD reference embedded in implementer prompts
```
