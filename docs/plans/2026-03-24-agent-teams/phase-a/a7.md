# A7: Update implementer-prompt.md for teammate self-management

**Avoid:** Don't make the implementer prompt too long — it's injected into every teammate's context. The current prompt is ~570 words. Target under 800 words. Teammates self-manage lifecycle now, but the prompt should describe what to do, not exhaustively how agent teams work.

**Avoid:** Don't duplicate the TDD reference (tdd.md) — keep the existing `**See:** skills/orchestrate/tdd.md` pattern.

## Steps

### Step 1: Read current implementer-prompt.md

Read `skills/orchestrate/implementer-prompt.md` end-to-end. Note:
- Current dispatch model: Agent tool dispatched by phase dispatcher
- No lifecycle management (dispatcher does that)
- No deviation rules (dispatcher handles those)
- No completion notes (dispatcher writes single completion.md)
- `Work from: [directory]` placeholder — not needed with auto-worktree

### Step 2: Rewrite the prompt

The new implementer prompt template needs these additions:
1. **Lifecycle management** — teammate marks own task in-progress/complete
2. **Per-task completion notes** — teammate writes `{PHASE_DIR}/{task_id_lower}-completion.md`
3. **Deviation rules 1-3** — auto-fix bugs, add critical validation, fix blockers
4. **Deviation rule 4** — send message to lead via mailbox for architectural changes
5. **No "Work from"** — teammate uses auto-provisioned worktree

Variables that the lead substitutes before dispatching the teammate:
- `{TASK_ID}` — the task ID (e.g., A1)
- `{TASK_METADATA}` — JSON task object from plan.json
- `{TASK_PROSE}` — content of the task .md file
- `{PLAN_DIR}` — absolute path to plan directory
- `{PHASE_DIR}` — absolute path to phase directory

New prompt structure:

```markdown
# Implementer Teammate Prompt Template

Use this template when spawning an implementer teammate. Substitute all {VARIABLES} before spawning.

**Variables:**
- `{TASK_ID}` — the task ID (e.g., A1)
- `{TASK_METADATA}` — JSON task object from plan.json
- `{TASK_PROSE}` — content of the task .md file
- `{PLAN_DIR}` — absolute path to plan directory
- `{PHASE_DIR}` — absolute path to phase directory

\```text
Teammate prompt:
  You are implementing {TASK_ID}

  ## Task Metadata (from plan.json)

  {TASK_METADATA}

  ## Task Instructions (from task file)

  {TASK_PROSE}

  ## Before You Begin

  Mark task in-progress:
  \```bash
  bash scripts/validate-plan --update-status {PLAN_DIR}/plan.json --task {TASK_ID} --status in_progress
  \```

  If you have questions about requirements, approach, or anything unclear — ask now.

  ## Your Job

  1. Follow TDD for all implementation — write failing test, verify FAILS, write minimal code, verify PASSES, refactor, commit. **See:** `skills/orchestrate/tdd.md`
  2. If this task consumes output from a prior task, write a boundary integration test using real components
  3. Implement exactly what the task specifies
  4. Verify implementation works
  5. Commit your work
  6. Self-review (see below)
  7. Write completion notes (see below)
  8. Mark task complete and report back

  **While you work:** If you encounter something unexpected, ask questions. Don't guess.

  ## Deviation Rules

  | Rule | Trigger | Action |
  |------|---------|--------|
  | 1: Auto-fix bug | Code doesn't work as intended | Fix it, document in completion notes |
  | 2: Auto-add critical | Missing validation, auth, error handling | Add it, document |
  | 3: Auto-fix blocker | Missing dep, broken import, wrong types | Fix it, document |
  | 4: STOP | Architectural change (new table, library swap, breaking API) | Send message to lead describing what change is needed and why. Wait for response before proceeding. |

  Only fix issues caused by your task. Pre-existing issues: document and move on.

  ## Self-Review

  Before reporting back, review your work:
  - Did I implement everything in the spec? Miss any requirements?
  - Is code clean, names clear, following codebase patterns?
  - Did I avoid overbuilding (YAGNI)?
  - Do tests verify behavior (not mocks)? Did I follow TDD?
  - If cross-task boundaries, did I write boundary integration tests?

  Fix any issues found during self-review.

  ## Completion Notes

  Write to {PHASE_DIR}/{task_id_lower}-completion.md:

  \```markdown
  # {TASK_ID} Completion Notes

  **Summary:** [2-3 sentences: what was built]
  **Deviations:** [Each: what changed, which rule, reason. "None" if plan followed exactly.]
  **Files changed:** [list]
  **Test results:** [pass/fail counts]
  \```

  ## Mark Complete

  \```bash
  bash scripts/validate-plan --update-status {PLAN_DIR}/plan.json --task {TASK_ID} --status complete
  \```

  ## Report Format

  When done, report:
  - What you implemented
  - What you tested and test results
  - Files changed
  - Self-review findings (if any)
  - Deviations (if any)
  - Any concerns
\```
```

Note: The actual file should have proper fenced code blocks without the backslash escapes — those are shown here to avoid nesting issues in the plan prose.

### Step 3: Verify the prompt is complete

Check that the prompt covers all migrated responsibilities from the design doc's "Migrated Responsibilities" table:
- Task lifecycle (mark in-progress/complete) — Steps "Before You Begin" + "Mark Complete"
- Completion notes — "Completion Notes" section
- Deviation rules 1-3 — "Deviation Rules" table
- Deviation rule 4 — "Deviation Rules" table (send message to lead)
