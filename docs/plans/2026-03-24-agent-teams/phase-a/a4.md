### Step 2: Write the new SKILL.md

The new structure should include these sections in order:

1. **Frontmatter** — same name and description
2. **Header** — "Execute plans via agent teams. Phases run sequentially; tasks within each phase run in parallel as teammates. Push-based idle notifications replace polling."
3. **Requires block** — `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`
4. **Prompt Templates** — 3-row table (implementer, task-reviewer, implementation-review). No phase-dispatcher row.
5. **Progress Tracking** — TaskCreate per phase, simplified from current
6. **Setup** — verify env var, read workflow, count phases, validate schema, set status, record PLAN_BASE_SHA, push branch
7. **Per-Phase Execution (Sequential)** — with subsections:
   - Prepare phase (worktree, bootstrap, handoff notes)
   - Spawn implementer teammates (parallel, one per task)
   - Process completions (push-based idle notifications, dispatch reviewer teammates)
   - Handle escalations (mailbox messages from teammates)
   - Phase wrap-up (merge task branches, impl-review, Review Loop Protocol, status update, phase PR for multi-phase)
8. **Review Loop Protocol** — unchanged from current
9. **Single-Phase Plans** — simplified (no integration branch, no phase PR)
10. **After All Phases (Multi-Phase Only)** — unchanged
11. **Key Constraints** — add env var check and schema validation rows
12. **Integration** — unchanged

Key details for the Per-Phase Execution section:
- Spawn teammates only for tasks with no unmet dependencies (use `validate-plan --check-deps` to gate)
- Each task spawned as a teammate with its own auto-worktree from the current feature branch
- Lead receives idle notifications when teammates finish (no sleep/poll)
- **Teammate Lifecycle Protocol**: when implementer goes idle, lead dispatches reviewer teammate. If reviewer finds issues, lead sends review feedback to the *original implementer* via mailbox messaging (implementer still has files in context). Implementer fixes and goes idle again. Loop until review passes. Lead validates with `validate-plan --criteria`. Lead kills teammate only after everything passes.
- **Incremental merge**: after each task passes review and teammate is killed, lead immediately merges that task's branch into the feature branch. This ensures dependent tasks see prerequisite code in their worktrees.
- **Dependency gate**: when a task's dependencies resolve, lead runs `validate-plan --check-deps plan.json --task {TASK_ID}` to verify all dependency tasks have status `complete` before spawning a new teammate.
- **Phase Completion Gate**: lead cannot mark a phase complete until ALL teammates for that phase (implementers and reviewers) are terminated. This prevents advancing while teammates have unresolved work.
- After all teammates killed: dispatch implementation-review
- For multi-phase: create phase PR, external review gate, merge, clean up
- Cross-phase handoff notes written by lead between phases
